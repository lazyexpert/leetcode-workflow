# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A **Claude Code plugin marketplace** that ships a single bundled plugin (`leetcode-workflow`) containing six skills: `init`, `new`, `done`, `retry`, `abort`, `update`. End users install the marketplace, run `/leetcode-workflow:init` in an empty directory to scaffold a personal practice repo, then use the rest of the skills (`/leetcode-workflow:new`, `:done`, `:retry`, `:abort`, `:update`) inside that repo.

The skills are tightly coupled — shared DB, shared views, shared lifecycle. They ship as one plugin so the user installs once and gets the full toolkit. The `/leetcode-workflow:<skill>` invocation form is the only supported one.

**This repo contains only the plugin code.** User data — problems, solve history, the SQLite DB, the Markdown views — lives in each user's own practice repo, generated by `/leetcode-init`. There is no fork-and-own model; users get the latest plugin code via Claude Code's marketplace update mechanism, and they own their data forever.

## Distribution model

Claude Code has **no central plugin registry**. Plugins live in marketplaces — git repos with a `.claude-plugin/marketplace.json` manifest. Users add marketplaces via:

```
/plugin marketplace add lazyexpert/leetcode-workflow
/plugin install leetcode-workflow@leetcode-workflow
```

Once installed, the skills become available across all the user's projects. The "name claim" is just owning the GitHub repo — there's no other namespace to coordinate with. Eventually we may submit to Anthropic's official marketplace (via [claude.ai/settings/plugins/submit](https://claude.ai/settings/plugins/submit)), but that's a v1+ concern.

### Planned layout

```
.claude-plugin/
  marketplace.json                 # marketplace manifest (lists the one plugin)
plugins/
  leetcode-workflow/
    .claude-plugin/plugin.json     # plugin manifest (lists the six skills)
    skills/
      init/SKILL.md
      new/SKILL.md     + scripts/{fetch.py, scaffold.py}
      done/SKILL.md    + scripts/done.py
      retry/SKILL.md   + scripts/retry.py
      abort/SKILL.md   + scripts/abort.py
      update/SKILL.md  + scripts/update.py
    lib/                           # shared by all skills in the plugin
      db.py                        # SQLite helpers, config loading
      render.py                    # MD view generators
    schema-baseline.sql            # FROZEN v0 schema, never edited after release
    migrations/
      0001_<desc>.sql
      0002_<desc>.sql
      ...
README.md
CLAUDE.md
LICENSE
```

The marketplace contains exactly one plugin (`leetcode-workflow`); skills live underneath it. Cross-skill imports of `db`/`render` use `sys.path.insert` to the plugin root (`plugins/leetcode-workflow/lib/`).

The user's practice repos are entirely separate — they live wherever `/leetcode-init` was run. Their layout (created by init):

```
.claude/
  practice.db               # gitignored
  practice.sql              # committed; deterministic dump
.gitignore
config.json
src/{Easy,Medium,Hard,SQL}/<number>.<Title_With_Underscores>/
  README.md
  solution.<ext>
progress.md                 # generated view
timings.md                  # generated view
retry.md                    # generated view
patterns-coverage.md        # generated view
history.md                  # generated view
```

Nothing from this marketplace repo ends up in their working tree.

## Architecture: DB as source of truth, Markdown as views

User practice repos store all structured state in `.claude/practice.db` (SQLite, gitignored). The five root-level Markdown files are **generated views**, regenerated from the DB on every skill invocation. Users never edit them by hand — the next skill overwrites manual changes wholesale.

`.db` is gitignored (binary, doesn't diff cleanly). What lands in git is `.claude/practice.sql`, a deterministic `.dump` produced after every mutation. The init marker is `practice.sql` — every initialised repo has it; missing → not a leetcode-workflow repo. Skills auto-rebuild `.db` on first run if `.sql` is present but `.db` is missing (typical fresh-clone state).

### Tables

| Table | Holds |
|---|---|
| `problems`   | stable metadata: `number PK, title, difficulty, kind, folder, created_at` |
| `attempts`   | one row per solve session: `id PK, problem_number FK, started_at, duration_minutes, revisit`. `duration_minutes IS NULL` while in progress. |
| `patterns`   | `problem_number FK, pattern, created_at` — replaced wholesale per `/leetcode-done` |
| `thresholds` | `difficulty PK, minutes` — mirrors `config.json: retry_thresholds_minutes` |
| `settings`   | singleton key/value bag: `review_cooldown_days`, `schema_version`, `plugin_version_seen` |

### Views

- `retry_flags` — algorithmic problems with three boolean flag columns (`timing_bad`, `complexity_bad`, `stale`) computed from the latest attempt + thresholds + cooldown. Renderer joins set flags with `+` (e.g. `stale`, `complexity+stale`, `timing+complexity+stale`) for `retry.md`.

## Skills

All invoked as `/leetcode-workflow:<skill>`. No aliases — namespaced shortcuts like `/lc-new` won't survive the plugin namespacing cleanly, so the canonical form is the only form.

| Skill | What it does |
|---|---|
| `/leetcode-workflow:init` | Bootstrap a fresh practice repo at `cwd`. Refuses if `cwd` contains anything other than `.git`. Runs `git init` if needed, creates `.claude/`, `config.json`, empty MD views, `src/{Easy,Medium,Hard,SQL}/`, applies `schema-baseline.sql` + all migrations. |
| `/leetcode-workflow:new <url>` | Scaffold a problem from a LeetCode URL. Pipes `fetch.py` (GraphQL → manifest) into `scaffold.py`. Reiteration mode (truncate + new attempt) if the folder already has content. |
| `/leetcode-workflow:done` | Close out the current attempt. Detects the modified non-empty `solution.<ext\|sql>` under `src/`. Classifies via the `claude` CLI. Reports timing verdict (within/over threshold), patterns, complexity verdict (optimal/revisit). Auto-commits with `{N}. {Easy\|Medium\|Hard\|SQL}. {Title}`. |
| `/leetcode-workflow:retry [N]` | Pick a problem to revisit. No arg → random from `retry_flags WHERE stale = 1` (cooldown elapsed). With arg → explicit pick (cooldown bypassed). Strips body to a signature-only template via `claude` so the LC judge signature is preserved for the rewrite. Does **not** commit. |
| `/leetcode-workflow:abort` | Drop the latest in-progress attempt. Sole-attempt for the problem (typical aborted `/leetcode-workflow:new`) → full rollback: removes the folder, deletes problem + patterns + attempt rows. Otherwise → restore the solution file from `HEAD` via `git checkout`. Does **not** commit. Destructive on uncommitted edits in the affected file. |
| `/leetcode-workflow:update` | Apply pending DB migrations after a plugin update. Reads `settings.schema_version`, runs each `migrations/000N_*.sql` with N greater, bumps version. Re-renders views. |

Every skill except `init` starts by validating that `cwd` is a leetcode-workflow repo — checks `.claude/practice.sql` exists. If not, errors with: *"Not a leetcode-workflow repo. Run /leetcode-workflow:init or cd into one."*

## Configuration

User-editable `config.json` at the practice repo root:

```json
{
  "language":            { "extension": "ts", "name": "typescript" },
  "retry_thresholds_minutes": { "Easy": 15, "Medium": 30, "Hard": 60 },
  "review_cooldown_days": 7,
  "patterns": ["Two Pointers", "Sliding Window", "Binary Search", "..."]
}
```

| Key | Effect |
|---|---|
| `language.extension` | Filename suffix for algorithmic solutions (`solution.<ext>`). SQL is always `.sql`. |
| `language.name` | Code-fence language hint passed to the classifier prompt. Affects classification quality, nothing else. |
| `retry_thresholds_minutes` | Solve time past which `timing_bad = 1`. |
| `review_cooldown_days` | Days since last attempt after which `stale = 1`. Also the minimum age for `/leetcode-retry`'s random picks. |
| `patterns` | Closed enum of classifier labels + render order for `patterns-coverage.md`. Defaults to 18 common LC patterns. Add niche ones (Union Find, Line Sweep, Segment Tree) or trim for coarser buckets. Classifier responses outside this list are filtered with a warning — no junk rows in the DB. |

`config.json` is the user-editable surface; relevant values are mirrored into the `thresholds` and `settings` tables on every skill invocation so the `retry_flags` VIEW can reference them at query time.

## Migration strategy

- **Frozen baseline.** `schema-baseline.sql` is the v0 schema, never edited after release. Just `CREATE TABLE`s + indexes + the initial `retry_flags` view + a single `INSERT INTO settings ('schema_version', '0')`.
- **Numbered migrations.** `migrations/0001_<desc>.sql`, `0002_<desc>.sql`, etc. Each wrapped in `BEGIN; ... COMMIT;` for atomicity. Last statement in every migration: `INSERT OR REPLACE INTO settings (key, value) VALUES ('schema_version', '<N>');`.
- **Schema version in `settings`, not `PRAGMA user_version`.** Reason: `sqlite3 .dump` does *not* preserve `user_version`, so a teammate cloning a practice repo and rebuilding `.db` from `.sql` would land at version 0 and re-run already-applied migrations. The `settings` table IS included in the dump — survives the round-trip.
- **Single code path.** `/leetcode-workflow:init` and `/leetcode-workflow:update` both run the same migration runner. Init seeds the baseline first. Both end users at the latest version.
- **Update nudge.** The plugin's `plugin.json` ships a `version` field. After every skill, compare against `settings.plugin_version_seen`; if different, print one trailing line: `ⓘ leetcode-workflow updated to vN — run /leetcode-workflow:update to apply migrations`. Bump the seen-version on successful `/leetcode-workflow:update`.

## Pedagogical contract

Hard rule, enforced in every skill's SKILL.md:

> **Never produce solution code unless explicitly asked.** All discussion stays in hint form — point out time/space complexity of the current approach, suggest there is a more efficient data structure, name the algorithmic pattern that fits, or ask a guiding question. Only write or complete `solution.<ext>` / `solution.sql` when the user directly requests it.

Claude is configured as a coach, not a solution generator. The classifier itself only receives the user's already-written code; it never produces new code.

## Conventions

- **Python 3.10+** required for scripts (PEP 604 union syntax). `from __future__ import annotations` defensively in each script for 3.9 tolerance.
- **No external Python dependencies.** stdlib + `sqlite3` CLI for dumps. Keep it that way — install friction matters.
- **macOS-tested.** Linux should work; Windows probably needs minor path tweaks.
- **Git messages from `/leetcode-done`** are fixed: `{N}. {Easy|Medium|Hard|SQL}. {Title}`. Not configurable.
- **Folder naming.** `<number>.<Title_With_Underscores>` under `src/<Easy|Medium|Hard|SQL>/`. Underscores, not spaces.
- **Cross-skill imports.** Skills import `db` and `render` from `lib/` via a `sys.path.insert` to the plugin root. Plugin layout is stable, so this works.
