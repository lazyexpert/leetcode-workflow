# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A **Claude Code plugin marketplace** that ships a single bundled plugin (`leetcode-workflow`) containing seven slash commands: `init`, `new`, `pick`, `done`, `retry`, `abort`, `update`. End users install the marketplace, run `/leetcode-workflow:init` in an empty directory to scaffold a personal practice repo, then use the rest of the commands inside that repo.

The commands are tightly coupled â€” shared DB, shared views, shared lifecycle. They ship as one plugin so the user installs once and gets the full toolkit. The `/leetcode-workflow:<command>` invocation form is the only supported one.

**This repo contains only the plugin code.** User data â€” problems, solve history, the SQLite DB, the Markdown views â€” lives in each user's own practice repo, generated by `/leetcode-workflow:init`. There is no fork-and-own model; users get the latest plugin code via Claude Code's marketplace update mechanism, and they own their data forever.

## Status

**Pre-release v0.1.0.** Schema is frozen against `schema-baseline.sql`; future shape changes ship as numbered migrations. The command API is settled; minor rough edges may still surface as more users dogfood it.

## Distribution model

Claude Code has **no central plugin registry**. Plugins live in marketplaces â€” git repos with a `.claude-plugin/marketplace.json` manifest. Users add marketplaces via:

```
/plugin marketplace add lazyexpert/leetcode-workflow
/plugin install leetcode-workflow@leetcode-workflow
```

Once installed, the seven commands become available across all the user's projects. The plugin is self-publishable: any user running `/plugin marketplace add lazyexpert/leetcode-workflow` installs it directly from this repo's public `marketplace.json` â€” no Anthropic-side submission required. Curated discovery via Anthropic's web UI may come later; that's a v1+ concern.

### Repo layout

```
.claude-plugin/
  marketplace.json                 # marketplace manifest (lists the one plugin)
plugins/
  leetcode-workflow/
    .claude-plugin/plugin.json     # plugin manifest (name + version)
    commands/                      # one file per slash command â€” orchestration prose

      init/init.py
      new/{fetch.py, detect_reiteration.py, scaffold_new.py}
      pick/{choose_mode.py, coverage_gaps.py}
      done/{detect_problem.py, record_attempt.py, commit.py}
      retry/pick_problem.py
      abort/abort.py
      update/update.py
    lib/                           # shared modules + shared CLI scripts
      db.py                        # SQLite helpers, config loading
      render.py                    # MD view generators
      migrate.py                   # migration runner (used by init + update)
      plugin_meta.py               # reads plugin.json version
      nudge.py                     # update-nudge banner (CLI, called from every command)
      apply_solution_template.py   # CLI: writes signature + opens attempt (used by new+retry+pick)
      render_and_dump.py           # CLI: regenerate views + dump SQL
```

The marketplace contains exactly one plugin. Skill scripts under `scripts/<name>/` import `db`/`render`/etc. from `lib/` via `sys.path.insert(0, str(Path(__file__).resolve().parents[2] / 'lib'))`.

`commands/<name>.md` is the canonical orchestration: a slash-command file with frontmatter (`description`, `allowed-tools`) and a body of step-by-step instructions for the model. The body invokes scripts via `${CLAUDE_PLUGIN_ROOT}/scripts/<name>/<script>.py` and `${CLAUDE_PLUGIN_ROOT}/lib/<script>.py`.

The user's practice repos are entirely separate â€” they live wherever `/leetcode-workflow:init` was run. Their layout (created by init):

```
.claude/
  practice.db               # gitignored
  practice.sql              # committed; deterministic dump
.gitignore
README.md                   # workflow + config reference, scaffolded by init
config.json
src/{Easy,Medium,Hard,SQL}/<number>.<Title_With_Underscores>/
  README.md
  solution.<ext>
progress.md                 # generated view
timings.md                  # generated view
retry.md                    # generated view
patterns-coverage.md        # generated view
history.md                  # generated view
```

Nothing from this marketplace repo ends up in their working tree.

## Architecture: DB as source of truth, Markdown as views

User practice repos store all structured state in `.claude/practice.db` (SQLite, gitignored). The five root-level Markdown files are **generated views**, regenerated from the DB on every command invocation. Users never edit them by hand â€” the next command overwrites manual changes wholesale.

`.db` is gitignored (binary, doesn't diff cleanly). What lands in git is `.claude/practice.sql`, a deterministic `.dump` produced after every mutation. The init marker is `practice.sql` â€” every initialised repo has it; missing â†’ not a leetcode-workflow repo. Commands auto-rebuild `.db` on first run if `.sql` is present but `.db` is missing (typical fresh-clone state).

### Tables

| Table | Holds |
|---|---|
| `problems`   | stable metadata: `number PK, title, difficulty, kind, folder, created_at` |
| `attempts`   | one row per solve session: `id PK, problem_number FK, started_at, duration_minutes, revisit`. `duration_minutes IS NULL` while in progress. |
| `patterns`   | `problem_number FK, pattern, created_at` â€” replaced wholesale per `/leetcode-workflow:done` |
| `thresholds` | `difficulty PK, minutes` â€” mirrors `config.json: retry_thresholds_minutes` |
| `settings`   | singleton key/value bag: `review_cooldown_days`, `schema_version`, `plugin_version_seen` |

### Views

- `retry_flags` â€” algorithmic problems with three boolean flag columns (`timing_bad`, `complexity_bad`, `stale`) computed from the latest attempt + thresholds + cooldown. Renderer joins set flags with `+` (e.g. `stale`, `complexity+stale`, `timing+complexity+stale`) for `retry.md`.

## Commands

All invoked as `/leetcode-workflow:<name>`.

| Command | What it does |
|---|---|
| `/leetcode-workflow:init` | Bootstrap a fresh practice repo at `cwd`. Refuses if `cwd` contains anything other than `.git`. Asks two short questions (language, retry thresholds). Runs `git init` if needed, creates `.claude/`, `config.json`, `README.md`, empty MD views, `src/{Easy,Medium,Hard,SQL}/`, applies `schema-baseline.sql` + all migrations. |
| `/leetcode-workflow:new <url>` | Scaffold a problem from a LeetCode URL. `fetch.py` writes the manifest (incl. LC's per-language signature template) to `/tmp`; `scaffold_new.py` creates the folder and seeds `solution.<ext>` with the signature. Reiteration mode (existing non-empty solution â†’ strip body via the model + open new attempt) when the folder already has content. |
| `/leetcode-workflow:pick` | "What should I solve next?". Rolls against `pick_retry_ratio`; on `retry` runs the retry flow, on `new` reads coverage gaps and asks the model to suggest a fresh problem URL targeting under-covered patterns, then runs the new flow. |
| `/leetcode-workflow:done` | Close out the current attempt. Detects the modified non-empty `solution.<ext\|sql>` under `src/`. Model classifies the solution against the closed pattern enum (prompt inline in `commands/done.md`). Reports timing verdict, patterns, complexity verdict. Auto-commits with `{N}. {Easy\|Medium\|Hard\|SQL}. {Title}`. |
| `/leetcode-workflow:retry [N]` | Pick a problem to revisit. No arg â†’ random from `retry_flags WHERE stale = 1` (cooldown elapsed). With arg â†’ explicit pick (cooldown bypassed). Model strips the previous solution to a signature-only template; `apply_solution_template.py` writes it. Does **not** commit. |
| `/leetcode-workflow:abort` | Drop the latest in-progress attempt. Sole-attempt for the problem (typical aborted `/new`) â†’ full rollback: removes the folder, deletes problem + attempts (CASCADE wipes patterns). Otherwise â†’ restore the solution file from `HEAD` via `git checkout`. Does **not** commit. Destructive on uncommitted edits in the affected file. |
| `/leetcode-workflow:update` | Apply pending DB migrations after a plugin update. Reads `settings.schema_version`, runs each `migrations/000N_*.sql` with N greater. Atomic: a failed migration rolls back via `conn.rollback()`. Bumps `plugin_version_seen` to the manifest's current version, dismissing the nudge. |

Every command except `init` starts by validating that `cwd` is a leetcode-workflow repo â€” the lib's `db.open_db()` raises `NotInitialized` if neither `practice.db` nor `practice.sql` exists. Each command's last step runs `lib/nudge.py` to surface the update banner if the plugin has been updated since the user's last `/update`.

## Configuration

User-editable `config.json` at the practice repo root:

```json
{
  "language":            { "extension": "ts", "name": "typescript" },
  "retry_thresholds_minutes": { "Easy": 15, "Medium": 30, "Hard": 60 },
  "review_cooldown_days": 7,
  "pick_retry_ratio":     0.0,
  "patterns": ["Two Pointers", "Sliding Window", "Binary Search", "..."]
}
```

| Key | Effect |
|---|---|
| `language.extension` | Filename suffix for algorithmic solutions (`solution.<ext>`). SQL is always `.sql`. Also drives which LC code snippet is fetched as the signature template. |
| `language.name` | Code-fence language hint for the classifier prompt + LC `langSlug` lookup (with aliases for `go â†’ golang`, `python â†’ python3`). |
| `retry_thresholds_minutes` | Solve time past which `timing_bad = 1`. |
| `review_cooldown_days` | Days since last attempt after which `stale = 1`. Also the minimum age for `/leetcode-workflow:retry`'s random picks. |
| `pick_retry_ratio` | 0..1 share of `/leetcode-workflow:pick` invocations that route to the retry pool instead of suggesting a fresh problem. Default 0.0 (always new). |
| `patterns` | Closed enum of classifier labels + render order for `patterns-coverage.md`. Defaults to 18 common LC patterns. Add niche ones (Union Find, Line Sweep, Segment Tree) or trim for coarser buckets. Classifier responses outside this list are filtered with a warning â€” no junk rows in the DB. |

`config.json` is the user-editable surface; relevant values are mirrored into the `thresholds` and `settings` tables on every command invocation so the `retry_flags` VIEW can reference them at query time.

## Migration strategy

- **Frozen baseline.** `schema-baseline.sql` is the v0 schema, never edited after release. `CREATE TABLE`s + indexes + the initial `retry_flags` view + seeds for `settings.schema_version='0'` and `settings.plugin_version_seen=''`.
- **Numbered migrations.** `migrations/0001_<desc>.sql`, `0002_<desc>.sql`, etc. Each wrapped in `BEGIN; ... COMMIT;` for atomicity. Last statement in every migration: `INSERT OR REPLACE INTO settings (key, value) VALUES ('schema_version', '<N>');`.
- **Schema version in `settings`, not `PRAGMA user_version`.** Reason: `sqlite3 .dump` does *not* preserve `user_version`, so a teammate cloning a practice repo and rebuilding `.db` from `.sql` would land at version 0 and re-run already-applied migrations. The `settings` table IS included in the dump â€” survives the round-trip.
- **Single code path.** `init.py` and `update.py` both call `lib/migrate.apply_pending(conn)`. Init seeds the baseline first, then applies migrations. Both end users at the latest version.
- **Atomic on failure.** If a migration's `executescript` raises, `apply_pending` issues a `conn.rollback()` to undo any DDL the migration ran before crashing. Earlier migrations in the same call stay applied; the failed migration leaves no partial state.
- **Update nudge.** `lib/nudge.py` compares `plugin.json: version` against `settings.plugin_version_seen` and prints `â“˜ leetcode-workflow updated to vN â€” run /leetcode-workflow:update to apply migrations` when they differ. Each command runs nudge as its last step. Init and update both write `plugin_version_seen` (init pre-populates so fresh users don't see a day-1 nudge).

## Pedagogical contract

Hard rule, enforced in every command's body:

> **Never produce solution code unless explicitly asked.** All discussion stays in hint form â€” point out time/space complexity of the current approach, suggest there is a more efficient data structure, name the algorithmic pattern that fits, or ask a guiding question. Only write or complete `solution.<ext>` / `solution.sql` when the user directly requests it.

Reinforced by data-flow constraints: `fetch.py` writes the LC manifest to `/tmp` (not stdout) and excludes the `topicTags` field, so neither the user's tool output nor the model's view leaks pattern hints. Commands explicitly instruct the model not to `cat` the manifest or read the scaffolded `README.md`.

## Conventions

- **Python 3.10+** required for scripts (PEP 604 union syntax). `from __future__ import annotations` defensively in each script for 3.9 tolerance.
- **No external Python dependencies.** stdlib + `sqlite3` CLI for `.dump`. Keep it that way â€” install friction matters.
- **No external Python dependencies in scripts; pytest is dev-only.**
- **macOS-tested.** Linux should work; Windows probably needs minor path tweaks.
- **Git messages from `/done`** are fixed: `{N}. {Easy|Medium|Hard|SQL}. {Title}`. Not configurable.
- **Folder naming.** `<number>.<Title_With_Underscores>` under `src/<Easy|Medium|Hard|SQL>/`. Underscores, not spaces.
- **Cross-script imports.** Skill scripts (`scripts/<name>/*.py`) import from `lib/` via `sys.path.insert(0, str(Path(__file__).resolve().parents[2] / 'lib'))`. Lib scripts (`lib/*.py`) import siblings via `sys.path.insert(0, str(Path(__file__).resolve().parent))`.
- **Scripts are deterministic.** No LLM calls inside scripts. Prompts that need the model (classify-solution, strip-solution-body, suggest-LC-URL) live inline in `commands/<name>.md`. The model produces output, the script consumes it as structured input.
- **JSON over the wire.** Scripts that need structured input read JSON from stdin or `--input <path>`. Multi-line / arbitrary-byte payloads (stripped solution bodies) use `--body-file <path>` to avoid shell-escaping fragility.

## Maintenance

### Adding a migration

1. Author `plugins/leetcode-workflow/migrations/000N_<short_description>.sql`. Wrap in `BEGIN; ... COMMIT;`. Last statement: `INSERT OR REPLACE INTO settings (key, value) VALUES ('schema_version', '<N>');` where N matches the file prefix.
2. Bump `plugins/leetcode-workflow/.claude-plugin/plugin.json: version`. Once released, users will see the update nudge on their next command and run `/leetcode-workflow:update` to apply.
3. Add a test in `tests/test_migrate.py` or extend an existing one. The runner accepts `migrations_dir=...` so tests can stub a fixture migration directory.

### Adding a command

1. Create `plugins/leetcode-workflow/commands/<name>.md` (frontmatter: `description`, `allowed-tools`).
2. Create `plugins/leetcode-workflow/scripts/<name>/` with the deterministic scripts.
3. Add tests under `tests/test_<name>_*.py`. Tests cover scripts only, not the command body.
4. Add the nudge invocation at the end of the command body so the update banner stays consistent across the suite.

### Running the test suite

```bash
pytest -q                          # all tests
pytest tests/test_db.py -v         # one file
pytest -k "atomicity"              # filter by name
```

`pytest.ini` sets `pythonpath = plugins/leetcode-workflow/lib` and `testpaths = tests`. `conftest.py` provides `practice_repo` (tmp dir + baseline applied + `db` reloaded), `empty_repo` (no init â€” for error-path tests), and `git_repo` (practice + git init + initial commit, for `commit.py` and `abort.py` git-restore tests).

### Linting

`ruff check .` runs the lint gate. Config in `ruff.toml`: line-length 100, rules `E/W` (pycodestyle), `F` (pyflakes), `I` (isort), `UP` (pyupgrade), `B` (bugbear), `SIM` (flake8-simplify). Per-file ignores cover tests (long assertion lines, fixture-first imports) and `init.py` (embeds the practice-repo README as a string literal â€” markdown line length is irrelevant).

Run locally:

```bash
pip install -r requirements-dev.txt
ruff check .                    # lint
ruff check . --fix              # auto-fix what's auto-fixable
```

CI invocation: `ruff check --output-format=github .` â€” emits inline annotations on PRs.

We do **not** enforce `ruff format` in CI. The codebase uses intentional alignment (`=` columns, `if x: foo()` one-liners with aligned colons) that the formatter would strip. Formatter adoption is a separate style decision.

### Coverage

Most tests run scripts as subprocesses, so default in-process coverage misses them. The setup measures both the parent pytest process and every subprocess it spawns, then combines.

How it works:
- `.coveragerc` at repo root holds the config. `branch = True`, `parallel = True`, no `[run] source` (it would resolve against subprocess cwd and break collection); we filter at report time via `[report] include`.
- `sitecustomize.py` at repo root calls `coverage.process_startup()` when `COVERAGE_PROCESS_START` is set in env. Python's site init runs it automatically if it sits on `sys.path`.
- `tests/conftest.py` detects `COVERAGE_PROCESS_START` and prepends repo root to `PYTHONPATH` so subprocesses pick up `sitecustomize`. It also pins `COVERAGE_FILE` to an absolute path so subprocess data files land in the repo root regardless of their cwd.

Run locally:

```bash
pip install -r requirements-dev.txt        # one-time

rm -f .coverage .coverage.*
COVERAGE_PROCESS_START="$PWD/.coveragerc" \
  coverage run --rcfile=.coveragerc -m pytest
coverage combine --rcfile=.coveragerc
coverage report --rcfile=.coveragerc
coverage html --rcfile=.coveragerc          # optional: htmlcov/index.html
```

Baseline as of this writing: **95% line coverage, 19 modules, 957 statements, 202 branches.** Lowest is `scripts/update/update.py` at 84% (only exercised when a real migration ships). Plain `pytest` (without `coverage run`) is unaffected â€” the scaffolding is fully gated on `COVERAGE_PROCESS_START`.

CI integration is straightforward: `pip install -r requirements-dev.txt` then the four commands above, ending with `coverage report --fail-under=<N>` when we set a gate. `coverage xml` produces a Codecov-compatible report.
