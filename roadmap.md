# Roadmap

Things to research and discuss later. Not committed scope.

- **Versioning + release hygiene.** Lint, pytest, and coverage CI are wired in `.github/workflows/test.yml` with a 95% coverage gate enforced via `.coveragerc` and a `**.md` paths-ignore so docs-only changes skip the matrix. Still open before v0.1.0 tag: define a release process (tag → bump `plugin.json: version` → push), and add a few light command wiring checks (e.g. every mutating command invokes `nudge.py`, every new command has matching script and `commands/<name>.md`, no dangling `${CLAUDE_PLUGIN_ROOT}` paths). Not blocking pre-release, but worth a clean story before users start installing.
- **Automated security scanning + badges.** Surface "this project takes security seriously" without asking users to read the code. Four pieces, all free for public repos:
  - **CodeQL** — GitHub-native SAST. One workflow file (`.github/workflows/codeql.yml`), runs on push/PR + weekly. Findings land in the Security tab. Yields a passing CI badge.
  - **ruff `S` rules** — add `"S"` to `select` in `ruff.toml`. Bandit-equivalent rules (subprocess `shell=True`, weak random, hardcoded creds, etc.) caught at PR time. Zero new tooling.
  - **OpenSSF Scorecard** — weekly workflow scores the repo's security posture (branch protection, signed releases, vuln response) on 0–10. The badge shows the live number — this is what users/registries actually check.
  - **GitHub built-ins** — secret scanning + Dependabot security alerts are free for public repos and on by default. Mention in README so users know they're active.
- **LC token in `config.json` to unlock premium problems.** `fetch.py` currently hits LeetCode's public GraphQL anonymously and returns exit code 2 on premium problems (paywalled `content`). Investigate accepting an authenticated session token (`LEETCODE_SESSION` cookie or similar) via `config.json` so premium problems can be scaffolded. Open questions: secure storage (probably `.claude/` gitignored, never `config.json` itself), token refresh model, what graceful-degradation looks like when the token expires.
- **Track solo vs hint-assisted solves.** At 100+ problems, "I solved it" stops being a useful signal — you forget which ones you owned end-to-end vs which ones you nudged Claude into hinting you through. Add a small dimension on `attempts` (likely an enum: `solo`, `hinted`, `gave_up`) and surface it in `retry.md` and `patterns-coverage.md` so the views answer "where am I actually shaky?" rather than just "where haven't I been recently?". Open questions: who labels (user self-reports at `/done`? Claude infers from the session? both with override?), default value, how to avoid making the tool feel judgy. Pairs naturally with the pedagogical contract — the tool already knows when hints were given, it just doesn't record it.
- **Authenticated submission to LeetCode without a browser.** Today the user solves locally and the plugin tracks timing/patterns/coverage, but the actual LC submission still happens in the browser. Investigate whether we can submit `solution.<ext>` directly via LC's API (same auth path as above) and capture the verdict (Accepted / Wrong Answer / TLE / runtime stats) into `attempts`. Would close the loop end-to-end inside Claude Code. Open questions: API stability/ToS, language→LC-language-id mapping, how to handle WA edge cases without surfacing test inputs (the pedagogical contract still applies).
- **Research charts option (mermaid?).** To reflect user's progress in README. Maybe daily/weekly trends growth, patterns coverage etc.
